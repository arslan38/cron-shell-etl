What Is It?

cron

  • Unix job scheduler (Vixie Cron, 1987)
  • Runs commands on a time-based schedule
  • Built into every Linux/macOS system
  • Zero installation, zero dependencies

shell (bash)

  • Bourne Again Shell (1989, GNU/GPL)
  • Glue language for Unix tools
  • Pipes, redirects, control flow
  • Direct access to awk, sed, psql, curl...
Crontab syntax: * * * * * command
| | | | +-- day of week   | | | +---- month   | | +------ day of month   | +-------- hour   +---------- minute

What Problem Does It Solve?

  • Scheduled automation: Run ETL jobs at fixed intervals without human intervention
  • Zero infrastructure overhead: No scheduler server, no web UI, no database backend needed
  • Universal availability: Pre-installed on every Unix system since 1987
Typical use case: A nightly cron job runs a shell script that extracts data from CSV files, transforms it with awk, and loads it into PostgreSQL -- a complete ETL pipeline in under 50 lines of shell code.

Before Airflow, Prefect, or Dagster existed,
cron + shell was the standard way to orchestrate data pipelines.

Trade-off: Simplicity and zero dependencies vs. observability and scalability.

Course Connection

  • YZV322E covers Apache Airflow as the primary orchestration tool
  • cron + shell is the historical predecessor that motivated Airflow's creation
  • Understanding cron's limitations explains why Airflow was invented:
Airflow was created at Airbnb in 2014 because cron jobs became unmanageable:
  • No visibility into pipeline state
  • No dependency graph between tasks
  • No built-in retry or alerting
  • Debugging = reading log files on remote servers

cron + shell occupies the "Scheduling" layer in the data engineering stack, doing the same job Airflow's scheduler does -- but with no abstraction over task dependencies.

Setup

Single command to build and run the entire pipeline:

docker-compose up --build

Dockerfile highlights

  • Base: ubuntu:22.04 (reliable cron)
  • Installs: cron, postgresql-client
  • Copies scripts + data + crontab
  • Entrypoint exports env vars for cron

docker-compose.yml

  • postgres: healthcheck ensures readiness
  • cron-worker: depends_on with condition
  • init.sql auto-creates tables on first run
  • Named volume for data persistence
Key pitfall solved: Cron does not inherit container environment variables. The entrypoint dumps printenv to /etc/environment so cron jobs can access POSTGRES_HOST, POSTGRES_PASSWORD, etc.

Repository Overview

File structure

cron-shell-etl/
+-- Dockerfile, docker-compose.yml
+-- entrypoint.sh, crontab
+-- data/sales_raw.csv
+-- scripts/
|   +-- pipeline.sh
|   +-- extract.sh
|   +-- transform.sh
|   +-- load.sh
+-- sql/init.sql

Design decisions

  • Watermark file: Tracks last processed row for incremental batch processing
  • Idempotency: ON CONFLICT DO UPDATE ensures safe re-runs
  • Error handling: set -euo pipefail stops pipeline on first error
  • Logging: etl_log table records every run's status and row count
  • Separation: Each E/T/L step is an independent script
